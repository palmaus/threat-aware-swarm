# Model Storage

Цель этого документа — не дать смешать код, временные артефакты и долгоживущие модели.

## Source of truth
- код, конфиги и docs живут в репозитории;
- долгоживущие модели и benchmark-артефакты живут в MLflow или внешнем хранилище;
- локальный `runs/` — рабочий буфер, а не постоянный registry.

## Что хранить в репозитории
- исходный код;
- Hydra-конфиги и experiment specs;
- документы и шаблоны отчётов;
- метаданные, достаточные для воспроизводимости.

## Что не хранить в репозитории
- крупные `*.zip` модели;
- TensorBoard events;
- временные `runs/*` артефакты;
- локальные tuning caches и диагностику, если они не нужны как эталон.

## Где хранить модели

| Вариант | Когда использовать |
| --- | --- |
| MLflow | основной runtime и registry поток |
| GitHub Release assets | публичные стабильные релизы |
| S3 / Drive / другое объектное хранилище | большие артефакты и внешняя раздача |
| Git LFS | только если реально нужен небольшой набор больших файлов внутри репо |

## Практические правила
- публикуй модель вместе с `run_id`, конфигом и минимальным описанием качества;
- для eval используй resolver (`mlflow:`, `clearml:`, локальный путь, alias `best|final|interrupt`), а не ручные ad-hoc пути;
- если модель идёт в demo/reporting, фиксируй её статус (`best_fair`, `best_privileged`, `hero_demo`).
